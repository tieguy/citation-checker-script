# Continuous Integration for citation-checker-script Design

> **Status (2026-05-01):** Implementation in progress on a contributor branch (`add-github-actions-ci`); not yet proposed upstream as a PR. The original draft assumed this work would stack on a `benchmark-source-availability` base that was later abandoned, so the implementation has been rebased onto current `main`. References below to `benchmark-source-availability`, `--dry-run`, source-availability schema fields, and an ~85-row dataset reflect the older base — substitute current `main` and the 156-row dataset when reading.

## Summary

This document describes a five-tier continuous integration (CI) system for `citation-checker-script`, a Wikipedia user script that uses AI language models to verify whether citations actually support the claims they are attached to. The project currently has no automated CI; this design introduces it from scratch.

The central challenge is that the codebase has two very different categories of correctness: deterministic infrastructure (does the code load, extract claims, fetch source text, and write output files correctly?) and probabilistic LLM accuracy (does the AI reach the right verdict, reliably, across providers?). A single "run the benchmark" job would be too slow and expensive for routine PR feedback, and too coarse-grained to distinguish an infrastructure bug from an accuracy regression. The design addresses this by splitting CI into five named tiers of increasing cost and scope. Tiers 0–2 are always free — they cover unit tests, the build consistency check, and the full non-LLM pipeline — and run on every PR including those from external contributors. Tiers 3 and 4 require API keys and real LLM calls; they are secrets-gated so they never expose credentials to forked PRs, and they run only on trusted branches or on a manual or scheduled trigger. A curated "smoke set" of 3–5 benchmark rows anchors the fast tiers, while a baseline snapshot of per-provider accuracy metrics lets tier 4 catch regressions against a known-good state rather than just eyeballing numbers. Implementation proceeds in seven sequential phases, from defining the smoke set through adding workflows and documentation, so each phase produces a testable deliverable before the next begins.

## Definition of Done

CI is "done" when:

- `.github/workflows/` contains workflows for five tiers (`unit-and-build`, `non-llm-infrastructure-sample`, `non-llm-infrastructure-full`, `llm-providers-sample`, `accuracy-regression`) wired to the agreed trigger map (PR / push-to-main / `workflow_dispatch` / scheduled).
- `benchmark/run_benchmark.js` gains a `--smoke-set` flag (filters to rows in `benchmark/smoke-set.json`). The existing `--dry-run` mode (from `benchmark-source-availability`, which this PR stacks on) is reused as-is for tiers 1 and 2.
- `benchmark/smoke-set.json` exists with 3–5 curated rows covering the four verdict classes (Supported / Partially Supported / Not Supported / Source unavailable), HTML / PDF / archive fetch types, and at least one must-pass entry.
- Top-level `package.json` has new npm scripts (canonical verbose names + short aliases).
- `benchmark/compare-to-baseline.js` compares a results file against `benchmark/baseline.json` with configurable per-provider accuracy tolerance and zero-tolerance on must-pass failures; exits non-zero on regression.
- `benchmark/baseline.json` is seeded from a one-time real benchmark run after secrets are configured (or stubbed with a clear marker until then).
- README documents the tier system, npm script usage, and how Alex enables tiers 3 and 4 (secrets list, baseline-seeding procedure).
- Tiers 0–2 run on every PR (including forked PRs) and every push to `main`; require no secrets.
- Tier 3 runs on push-to-`main` and trusted PRs (via `if: secrets.X != ''` skip pattern); skips cleanly on forked PRs.
- Tier 4 runs on `workflow_dispatch` and a weekly schedule; blocked on secrets until Alex flips them.
- The 78 existing unit tests on the `benchmark-source-availability` base continue passing; `npm run build -- --check` is clean; new comparator and smoke-set-filter tests pass.

## Glossary

- **CCS**: `citation-checker-script` — the browser-side Wikipedia user script that is the subject of this repository. Also the `npx ccs` CLI command added in factor-phase-2.
- **USync**: Wikipedia's "User Script Sync" mechanism that automatically propagates updates from a user's script page (`User:Alaexis/AI_Source_Verification`) to other wikis. Relevant because `main.js` is deployed via this path rather than through a build pipeline.
- **Wayback / archive fetch**: A source URL that resolves through the Internet Archive's Wayback Machine (`web.archive.org`). The benchmark distinguishes Wayback fetches from live HTML and PDF fetches because the proxy must strip Wayback preamble HTML before passing source content to the LLM.
- **must-pass**: A benchmark dataset entry tagged as a high-confidence ground-truth case that CI must never regress. The comparator applies zero tolerance to must-pass failures regardless of the overall accuracy tolerance setting.
- **smoke set / smoke-set.json**: A hand-curated list of 3–5 row IDs from the full benchmark dataset used by the fast tiers. Chosen to cover all four verdict classes, the three fetch types, and at least one must-pass entry; checked in so runs are deterministic.
- **dry run (`--dry-run`)**: A runner mode that skips the actual LLM provider call and emits a synthetic `"DRY_RUN"` verdict row instead. Everything else in the pipeline (dataset load, claim extraction, source fetch, results file) runs as normal. Used by tiers 1 and 2 to verify non-LLM infrastructure without incurring API costs.
- **factor-phase-1 / PR #118**: The first of three stacked PRs in the "factor-and-CLI" initiative. Extracts pure-logic functions from `main.js` into six `core/*.js` ESM modules, adds `scripts/sync-main.js` to keep `main.js` in sync, and introduces the first automated unit tests (`tests/`, 22 tests). Merged 2026-04-25, then reverted via PR #121, then re-applied as commit `ec4caf8` with conflict resolution merged via PR #123. PRs #119 and #120 (CLI and benchmark ESM refactor) merged on top of it the same day.
- **`benchmark-source-availability`**: A local branch that adds the source-availability schema and classifier (`core/availability.js`), an idempotent backfill script, deterministic runner short-circuit for unfetchable rows, the `npm run benchmark:dry-run` script, sandbox-row ingestion (8 new rows, 77→85 dataset), and 56 new tests (78 total across 13 files). To be rebased onto current main and opened upstream first; the CI PR stacks on top.
- **`core/` ↔ `main.js` consistency / `npm run build -- --check`**: `main.js` is a single-file user script; the `core/` modules are spliced into it between marker comments by `scripts/sync-main.js`. The `--check` flag exits non-zero if `main.js` is stale relative to `core/`. Tier 0 runs this check on every PR.
- **PublicAI / Apertus / Qwen**: Third-party AI provider services (accessed via the `publicai-proxy` worker) used as lower-cost LLM alternatives to Claude and OpenAI in benchmark runs. Apertus currently has a 5–10% API error rate that the comparator treats separately from verdict regressions.
- **Apertus error rate**: The observed rate at which the Apertus provider returns 5xx errors or timeouts rather than a verdict. The comparator reports these separately so a spike in provider errors does not mask or mimic an accuracy regression.
- **baseline / `benchmark/baseline.json`**: A snapshot of per-provider accuracy metrics (binary accuracy, must-pass failures, confusion-matrix diagonal) captured from a known-good benchmark run. The tier 4 comparator measures the current run against this baseline and fails CI if any metric falls below the configured tolerance.
- **confusion matrix diagonal floor**: Each cell on the main diagonal of the 4×4 verdict confusion matrix represents correct classifications for one verdict class. The baseline records a minimum floor for each diagonal cell; the comparator fails if any cell drops below its floor, catching class-specific regressions that aggregate accuracy might not surface.
- **binary accuracy**: A simplified accuracy metric that collapses the four verdict classes into two (correct / incorrect) for easier cross-provider comparison. Used as the primary regression signal in tier 4.
- **`pp` (percentage point)**: Unit used to express changes in accuracy — e.g. "+4.2 pp" means the accuracy score rose by 4.2 absolute percentage points, not 4.2% of the previous value.
- **`workflow_dispatch`**: A GitHub Actions trigger that lets a human manually start a workflow run from the GitHub UI or via the API, without a push or pull-request event.
- **secrets-gating (`if: secrets.X != ''`)**: A GitHub Actions job condition that causes a job to skip (rather than fail) when the required secret is absent — which is the case for all workflows triggered by pull requests from forks, where GitHub withholds secrets as a platform-level safety guarantee.
- **`actions/setup-node@v4`**: A standard GitHub-provided action that installs a specified Node.js version and optionally caches npm dependencies for faster workflow runs.
- **open-issues.md**: A local issue tracker (not a GitHub Issues list) at the workspace root spanning both sub-repos (`citation-checker-script` and `public-ai-proxy`). Referenced in the Additional Considerations section for items 1, 8, and 9.
- **verdict caching (open-issues.md item 8)**: A proposed optimization where benchmark result rows for unchanged prompt+source+provider combinations are cached between runs, avoiding redundant LLM calls. Not in scope for this design but called out as a future slot-in for tier 4's workflow structure.
- **on-wiki prompt hosting (open-issues.md item 1)**: A proposed future feature where the LLM system prompt lives on a Wikipedia user page rather than hardcoded in `main.js`, enabling prompt A/B testing by revision ID. CI (specifically a working tier 4 with a baseline) is listed as a prerequisite.

## Architecture

CI is structured as five complementary tiers, each verifying a distinct property of the system. Tiers run as separate GitHub Actions jobs (across two or three workflow files) so a failure in one does not cascade to the others.

### Tier composition

| Tier | Name | What it verifies | LLM | Secrets | Cost | Time |
|---|---|---|---|---|---|---|
| 0 | unit-and-build | unit tests + `core/` ↔ `main.js` consistency | – | – | free | ~5s |
| 1 | non-llm-infrastructure-sample | verifier pipeline on 3–5 rows | – | – | free | <5s |
| 2 | non-llm-infrastructure-full | verifier pipeline on full dataset (~85 rows) | – | – | free | ~30s |
| 3 | llm-providers-sample | provider auth + response parsing on 3–5 rows | yes | yes | ~$0.01 | ~30s |
| 4 | accuracy-regression | full benchmark vs baseline | yes | yes | ~$1–2 | ~25–30 min |

### Trigger map

| Event | Tiers run |
|---|---|
| Forked PR | 0, 1, 2 |
| Trusted PR (same-repo branch) | 0, 1, 2, 3 |
| Push to `main` | 0, 1, 2, 3 |
| `workflow_dispatch` | any tier (input parameter selects) |
| Scheduled (weekly) | 4 |

### Workflow file layout

- `.github/workflows/ci.yml` — tiers 0, 1, 2 (always-runnable, no secrets).
- `.github/workflows/llm-providers.yml` — tier 3 (secrets-gated).
- `.github/workflows/accuracy-regression.yml` — tier 4 (secrets-gated, manual + scheduled).

Splitting tier 3 and tier 4 into their own files keeps the trigger logic legible and lets contributors see at a glance which workflow is responsible for which kind of failure.

### Why tiers 1 and 2 are both included

Tier 1 (sample) targets pre-commit hooks, fast PR feedback, and forked-PR signal where speed matters. Tier 2 (full) catches dataset-wide breakage that the curated sample cannot. Both are free; the cost of running both is negligible. Splitting them lets contributors run the fast version locally and lets CI run both — they are complementary, not redundant.

### Forked-PR safety

GitHub Actions does not expose secrets to workflows triggered by PRs from forks (this is a platform-level safety guarantee, not something we configure). Tier 3 and tier 4 use an explicit secret-availability check (`if: ${{ secrets.PUBLICAI_API_KEY != '' }}`) and emit a clear "skipped — secrets unavailable" status rather than failing. This means external contributors can open PRs and still see tier 0/1/2 results without secrets being exposed to their workflow runs.

### Smoke-set selection

`benchmark/smoke-set.json` lists 3–5 row IDs from `benchmark/dataset.json` chosen to cover:
- At least one row per verdict class (Supported, Partially Supported, Not Supported, Source unavailable).
- HTML, PDF, and archive (Wayback) fetch types.
- At least one must-pass entry.

The exact row IDs are checked in so smoke runs are deterministic. The list is hand-edited when the dataset evolves; a stale smoke-set is caught the next time tier 1 or tier 3 runs (rows referenced in smoke-set.json must exist in dataset.json — the runner errors otherwise).

### Dry-run behavior (reused from `benchmark-source-availability`)

The `--dry-run` mode already exists in `run_benchmark.js` on the `benchmark-source-availability` base. It short-circuits the provider call and emits a synthetic result row:

```json
{
  "verdict": "DRY_RUN",
  "confidence": 0,
  "comments": "synthetic - dry run",
  "provider": "<provider-name>",
  "row_id": "<row-id>"
}
```

The rest of the pipeline (dataset load, claim extraction, source fetch via the proxy, results-file write, analyzer math) executes unchanged. This isolates non-LLM infrastructure for tier 1 and tier 2 verification. CI does not change the dry-run behavior — it only wires it into workflow jobs and composes it with the new `--smoke-set` flag for tier 1.

### Baseline format

`benchmark/baseline.json` is a snapshot of expected per-provider metrics:

```json
{
  "schema_version": 1,
  "captured_at": "YYYY-MM-DD",
  "captured_commit": "<sha>",
  "dataset_row_count": 76,
  "providers": {
    "claude-haiku-4-5": {
      "binary_accuracy": 0.857,
      "must_pass_failures": 0,
      "confusion_matrix_diagonal_floor": [12, 8, 14, 6]
    },
    "publicai-apertus-70b": { ... },
    "publicai-qwen": { ... }
  }
}
```

The compare script applies per-metric tolerance:
- Configurable percentage-point floor on `binary_accuracy` (default `-2pp`).
- Zero tolerance on `must_pass_failures` — any new failure fails CI.
- Diagonal-floor check on `confusion_matrix_diagonal_floor` — each diagonal must be ≥ baseline value.

Provider-API errors (timeouts, 5xx, malformed JSON) are reported separately and do not by themselves trigger a regression failure — see Additional Considerations on the Apertus error rate.

## Existing Patterns

Investigation found:

- **`citation-checker-script` has a `docs/design-plans/` convention.** Two prior plans exist: `2026-04-23-factor-and-cli.md` and `2026-04-24-benchmark-source-availability.md`. This plan follows the same convention.
- **No existing CI configuration.** The repo has no `.github/workflows/` directory; this design introduces it.
- **78 unit tests** across 13 files on the `benchmark-source-availability` base, runnable via `npm test` from the repo root. Tier 0 reuses these directly with no changes.
- **`npm run build -- --check`** (added on `factor-phase-1` / PR #118, merged) exits non-zero on `core/` ↔ `main.js` drift. Tier 0 reuses this directly.
- **`npm run benchmark:dry-run` already exists** on `benchmark-source-availability`. It skips LLM calls and emits synthetic result rows — exactly the primitive tiers 1 and 2 need. This design wires it into workflow jobs rather than building it from scratch.
- **`benchmark/` is ESM (post-#120, merged)** and imports `extractClaimText` from `core/claim.js`. The runner exports `shouldRunnerShortCircuit()` and emits structured results carrying `source_availability`, `runner_deterministic`, and per-row availability buckets — all available to the tier 4 comparator without changes.
- **`bin/ccs` CLI** (post-#119, merged) exposes the verifier as a command-line tool. Not directly used by CI, but available for ad-hoc developer smoke checks against a single citation.
- **Source-availability schema** (on `benchmark-source-availability` base): each dataset row carries `source_availability`, `observed_fetch`, `expected_tool_verdict`, and `human_reviewer_verdict`. Smoke-set selection criteria can use these structured fields directly.
- **The benchmark analyzer** already produces per-provider metrics, a 4×4 verdict confusion matrix, per-availability-class breakdowns, and a strict-mode must-pass check. Tier 4's baseline reuses these outputs rather than introducing new metrics.
- **`run_benchmark.js`** filters out rows with `needs_manual_review: true`. The smoke-set filter slots in at the same point in the pipeline.
- **No existing secrets-gating pattern.** This design introduces the `if: secrets.X != ''` skip pattern and documents it.

`public-ai-proxy` has a separate testing setup (PR #8 adds `npm test` against two `test/*.mjs` scripts). CI for that repo is a future follow-up and is not in scope here.

## Implementation Phases

### Phase 1: Smoke-set definition

**Goal:** Curate the 3–5 rows tier 1 and tier 3 use, with an explicit selection rationale.

**Components:**
- `benchmark/smoke-set.json` — list of row IDs with selection rationale (verdict class, fetch type, must-pass flag).
- `docs/benchmark-smoke-set.md` — documents the curation criteria and the procedure for updating the list when the dataset evolves.

**Dependencies:** None (uses existing `benchmark/dataset.json`).

**Done when:** `smoke-set.json` is checked in; criteria are documented; each criterion (verdict class coverage, fetch-type coverage, must-pass entry) is verifiably satisfied by the selected rows.

### Phase 2: Smoke-set filter for the runner

**Goal:** Add row-filtering support to `run_benchmark.js` so tier 1 and tier 3 process only the curated smoke set. The existing `--dry-run` mode is reused unchanged.

**Components:**
- `--smoke-set` flag in `benchmark/run_benchmark.js` — restricts processing to row IDs listed in `benchmark/smoke-set.json`. Composable with the existing `--dry-run` flag.
- Unit tests under `tests/` covering: filtered-row selection, error on missing smoke-set file, error when smoke-set references row IDs not present in the dataset, composition with `--dry-run`.

**Dependencies:** Phase 1 (smoke-set.json must exist).

**Done when:** `--smoke-set` works in isolation and in combination with `--dry-run`; new unit tests pass; total `npm test` count grows from 78 to ~82.

### Phase 3: NPM scripts wiring

**Goal:** Surface the tier 1, 2, 3 runner invocations as npm scripts with the agreed semantic names + short aliases. The existing `benchmark:dry-run` script (from `benchmark-source-availability`) is reused — these new scripts wrap it.

**Components:**
- Top-level `package.json` — add the verbose tier-aligned names (`benchmark:non-llm-infrastructure[:sample]`, `benchmark:llm-providers:sample`) and short aliases (`benchmark:infra[:sample]`, `benchmark:llm:sample`). The verbose names appear in CI workflow logs for self-documentation; the short aliases are for local use. Existing `benchmark:dry-run` is preserved.

**Dependencies:** Phase 2 (`--smoke-set` flag must exist).

**Done when:** all new scripts execute the correct underlying command; aliases redirect to canonical names; running each from a fresh checkout produces expected output without errors; the existing `benchmark:dry-run` continues to work.

### Phase 4: Tiers 0–2 workflow

**Goal:** Get the always-runnable tiers into GitHub Actions.

**Components:**
- `.github/workflows/ci.yml` — three jobs (`unit-and-build`, `non-llm-infrastructure-sample`, `non-llm-infrastructure-full`) running in parallel on `pull_request` and `push` to `main`. Node 20 LTS, `actions/setup-node@v4` with npm cache.

**Dependencies:** Phase 3 (npm scripts must exist).

**Done when:** workflow file is valid YAML; pushed to a feature branch, all three jobs run on a test PR and exit 0; required `npm ci` setup is correct; `actions/setup-node` cache works.

### Phase 5: Tier 3 workflow with secrets gating

**Goal:** Wire the LLM-providers smoke tier with safe forked-PR handling.

**Components:**
- `.github/workflows/llm-providers.yml` — single job `llm-providers-sample` with `if: ${{ secrets.PUBLICAI_API_KEY != '' }}` guard. Triggers: `pull_request`, `push` to `main`, `workflow_dispatch`.
- README section documenting the secrets list (`ANTHROPIC_API_KEY`, `PUBLICAI_API_KEY`, optional `GEMINI_API_KEY` / `OPENAI_API_KEY`) and forked-PR behavior.

**Dependencies:** Phase 3 (npm scripts), Phase 4 (workflow patterns established).

**Done when:** workflow runs on trusted PRs and skips cleanly on forked PRs; README documents secrets setup; manual `workflow_dispatch` works; no secrets are exposed to forked PR runs.

### Phase 6: Compare-to-baseline and tier 4 workflow

**Goal:** Wire the accuracy regression tier with baseline comparison.

**Components:**
- `benchmark/compare-to-baseline.js` — takes `results.json` and `benchmark/baseline.json`, applies per-provider tolerance, exits non-zero on regression. Exports a function for unit testing; CLI entry point for workflow use.
- `benchmark/baseline.json` — initial stub with `schema_version: 1` and a comment marker noting it must be seeded from a real run.
- `tests/compare_to_baseline.test.js` — unit tests for the comparator (tolerance behavior, must-pass strictness, diagonal-floor checks, schema-version handling).
- `.github/workflows/accuracy-regression.yml` — `workflow_dispatch` + weekly `schedule` triggers; runs `npm run benchmark` (full); pipes results into `compare-to-baseline.js`; uploads `results.json` and `analysis.json` as workflow artifacts.

**Dependencies:** Phase 5 (LLM tier patterns and secrets-gating idiom). Also depends on a one-time real benchmark run by Alex (or by us with Alex's approval) to seed `baseline.json` — out-of-band; can be a separate PR or a manual commit after secrets are flipped.

**Done when:** comparator unit tests pass; baseline schema is documented; workflow file is valid; manual `workflow_dispatch` produces artifacts; documented procedure for seeding baseline is clear.

### Phase 7: Documentation and handoff

**Goal:** Make the CI system maintainer-readable.

**Components:**
- README section on the tier system: what each tier verifies, when each tier runs, how to trigger manual runs, how to interpret each status check.
- Inline comments in workflow files explaining the trigger logic and secrets-gating idiom for future maintainers.
- A short note in the PR description (when this design ships as a PR) explaining the secrets that Alex needs to add for tiers 3 and 4 to activate.

**Dependencies:** Phases 1–6.

**Done when:** README accurately describes the system; a fresh reader can understand why each tier exists and when each runs without consulting this design plan; secrets-flip handoff to Alex is unambiguous.

## Additional Considerations

**Cost ceiling for tier 3.** Smoke-set runs are ~$0.01 each. A busy PR queue could rack up ~$1/week. Acceptable. If contributor activity grows enough that this matters, gating tier 3 behind a `benchmark` label (rather than firing on every trusted PR) is a one-line workflow change.

**Cost ceiling for tier 4.** ~$1–2 per run on the current ~85-row dataset. Weekly schedule = ~$50–100/year per provider × 3 providers = ~$150–300/year. If this is too high, the schedule can drop to bi-weekly or run only on push-to-`main` with significant changes (label-gated). Worth coordinating with Alex on cost expectations before turning the schedule on. Note: if PR #116 lands (expanding the dataset to 226 rows), this cost roughly triples — re-evaluate before activating the schedule.

**Verdict caching (open-issues.md item 8).** Caching would dramatically reduce tier 4 cost — re-runs of unchanged-prompt entries are free. Not in scope here, but tier 4's workflow should be structured so caching can be slotted in as a non-breaking enhancement (e.g., the runner reads/writes a cache file that the workflow uploads/downloads as an artifact).

**Apertus error rate (open-issues.md item 9).** Apertus has a 5–10% API error rate that pollutes regression-tier signal. The comparator treats provider errors as a separate category from verdict regressions and reports them but does not fail CI on errors alone. The runner can additionally retry once on 5xx/timeout (item 9's suggestion) to reduce the noise floor.

**Branch base.** This design plan stacks on `benchmark-source-availability`, which itself rebases onto current main (containing #117, #118, #119, #120 — all merged 2026-04-25). The CI PR is the next link in the stack; it cannot land until `benchmark-source-availability` is opened upstream and merged.

**Rebase risk on `benchmark-source-availability`.** PR #118's merge sequence overnight included a revert (#121) and a Claude-generated conflict-resolution PR (#123). Rebasing `benchmark-source-availability` onto current main may surface conflicts in `core/` and `benchmark/`. Resolve carefully and re-run the full test suite (`npm test`, `npm run build -- --check`, `npm run backfill:check`, `npm run sandbox:check`) before opening upstream. If conflicts are non-trivial, consider opening a draft PR first and asking Alex for direction.

**Path A vs Path B.** The original draft of this plan considered Path A (CI as a standalone PR off post-#118 main, duplicating the dry-run runner work that already exists on `benchmark-source-availability`). Path B (this plan — stack on `benchmark-source-availability`) was chosen because (a) it eliminates the duplication, (b) #119 and #120 also merged overnight so the prerequisite cost dropped to "rebase + open PR" rather than "wait and hope," (c) the CI PR becomes much smaller and more reviewable, and (d) Alex sees the source-availability and CI work in clear sequence rather than as two competing changes that both touch the runner.

**Connection to on-wiki prompt hosting (open-issues.md item 1).** CI is the prerequisite for that work. Once tier 4 is operational with a baseline, prompt-page edits become A/B-testable by revision ID against the same dataset — exactly the leverage that motivates moving the prompt on-wiki.
